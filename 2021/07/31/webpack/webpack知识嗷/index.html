# 1.Webpack 简介

Webpack 是一种前端资源构建工具，一个静态模块打包器，在 Webpack 看来，前端的所有资源文件（js/json/css/img/less）都会作为模块处理，它会根据模块的依赖关系进行静态分析，打包生成对应的静态资源（bundle）

# 1.1 Webpack 五个核心概念

  • Entry 入口指示 Webpack 以哪个文件为入口起点开始打包，分析构建内部依赖图
  • Output 输出指示，Webpack 打包后资源 bundles 输出到哪去，以及如何命名
  • Loader Loader 让 Webpack 能够去处理那些非 JS 文件（Webpack 自身只理解 JS）
  • Plugins 插件可以用于执行范围更广的任务，插件的范围包括，从打包优化和压缩，一直重新定义环境中的变量等
  • Mode 模式指示 Webpack 使用相应模式的配置，分为 development 模式和 production 模式
    1. development 会将 process.NODE_ENV 的值设为 development，启用 NamedChunksPlugin 和 NamedModulesPlugin，能让代码本地调式运行的环境
    2. production 会将 process.NODE_ENV 的值设为 production，启用 FlagDependencyUsagePlugin，FlagIncludedChunksPlugin,ModuleConcatenationPlugin,NoEmitOnErrorsPlugin,OccurrenceOrderPlugin,SideEffectsFlagPlugin 和 UglifyJsPlugin，他能让代码优化上线运行的环境

结论：

  • webpack 能处理 js/json 资源，不能处理 css/img 等其他资源
  • 生产环境和开发环境将 ES6 模块化编译成浏览器能识别的模块化
  • 生产环境比开发环境多一个压缩 js 代码

# 4 优化性能

# 4.1 热模块替换（HMR）hot module replacement --- 开发环境优化

  • 作用：一个模块发生变化，只会重新打包这一个模块（而不是打包所有模块），极大提升了构建速度
  • 使用：在 devServer 中开启 hot 为 true 就可以了

从三个方面入手：

  • 样式文件：可以使用 HMR 功能：因为 style-loader 内部实现了，所以开发环境下，使用 style-loader

  • js 文件：默认不能使用 HMR 功能：需要修改 js 代码，添加支持 HMR 功能的代码，而且只能对非入口文件的 js 文件进行处理

  • html 文件：默认不能使用 HMR 功能，同时会导致问题：html 文件不能热更新了，通过修改 entry 入口，将 html 文件引入可以解决不能热更新问题

# 4.2 source-map --- 开发环境优化

source-map 是一种提供源代码到构建后代码映射的技术（如果构建后代码出错了，通过映射可以追踪代码错误）

使用：在 webpack.config.js 中添加 devtools 配置项，他的值有：

[inline-| hidden-| eval-] [nosources-] [cheap- [module-] ] sourse-map

  • source-map：生成外部 map 文件，可以提供错误代码准确信息和源代码的错误位置
  • inline-source-map：生成一个内联 map 文件，可以提供错误代码准确信息和源代码的错误位置
  • hidden-source-map：生成外部 map 文件，可以提供错误代码错误原因，但没有位置，不能追踪源代码错误，只能提示到构建代码的错误位置
  • eval-source-map：生成多个内联 map 文件，可以提供错误代码准确信息和源代码的错误位置
  • nosources-source-map：生成外部 map 文件，可以提供错误代码的准确信息，但是没有任何源代码信息
  • cheap-source-map: 生成外部 map 文件，可以提供错误代码信息和源代码的错误信息（只能精确到行）
  • cheap-module-source-map：和 cheap-source-map 一样，module 会将 loader 的 source map 也引入

开发环境：要求速度快，调试更友好，所以选择 eval-source-map

生产环境：要看源代码需不需要隐藏以及调试是否友好，而且不能使用内联 map，因为会使文件代码体积变大，所以可以选择 source-map 或者 cheap-module-source-map

# 4.4 缓存

第二次构建时，会读取缓存，从而让速度更快一些

  • 使用 babel 缓存：在 preset 后面添加一个 cacheDirectory：true 即开启 babel 缓存（强缓存），当我们修改源代码的时候，浏览器不会请求服务器拿到我们修改过后的文件，而是继续使用缓存的文件
  • 文件资源缓存
    1. hash：在输出的文件名上面加上 [hash:10] 每次 webpack 构建时都会生成一个唯一的 hash 值，当我们重新打包的时候，就会根据版本号来重新请求服务器，但是我们只改一个文件也会导致所有的缓存重新请求服务器，导致缓存失效
    2. chunkhash：根据 chunk 生成的 hash 值，如果打包来源同一个 chunk，那么 hash 值就一样，但是 css 是 js 引进来的，同属于一个 hash 值，也不会改变
    3. contenthash：根据文件的内容生成的 hash 值，不同文件 hash 值一定不一样

# 4.11 dll

  • dll 是一个动态连接库，他会指示 webpack 哪些包是不参与打包的，dll 会对某些库进行单独打包，将多个库打包成 1 个 chunk
  • 作用：
    • node_modules 会被打包成一个 chunk，文件体积太大了，通过 dll 我们可以这些库单独拆开，打包成不同的 chunk
    • 只要打包了的库不变，我们永远只要运行一次就行，代码改变了我们只要反复运行 config 就行了

新建 webpack.dll.js 配置需要单独打包的库

在 webpack.config.js 中使用 webapck.DllReferencePlugin 告诉 webpack 哪些库不参与打包，同时是同时的名称也得变
