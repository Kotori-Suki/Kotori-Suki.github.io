# 1.node 是什么

1.Node.js 不是一门语言，不是库，不是框架，是一个构建于谷歌 V8 引擎之上的 js 运行时环境

2.Node.js 可以解析和执行 js 代码，可以完全脱离浏览器来运行

3.Node.js 中的 js 没有 DOM，BOM，在 Node 这个执行环境中为 js 提供了一些服务器级别的操作 API

  • 文件读写
  • 网络服务的构建
  • 网络通信
  • http 服务器

4.Node.js 使用事件驱动，非阻塞 IO 模型（异步）, 使他更加的轻量和高效

5.Node.js 包生态系统，npm 是世界上最大的开源库生态系统

# 5.ip 地址和端口号的概念

1. 所有联网的程序都需要进行网络通信，计算机中只有一个物理网卡，而且同一个局域网中，网卡的地址必须是唯一的

2. 网卡是通过唯一的 ip 地址来定位的，计算机只能识别 ip 地址，客户端需要通过 DNS 来解析成 IP 地址，得到一串（25.26.35.27）

3.ip 地址用来定位计算机，端口号用来定位具体的应用程序（所有需要联网通信的软件都必须具有端口号）

4. 端口号的范围从 0-65536 之间

5. 计算机的默认端口号最好不要去使用（http 服务的 80）

6. 我们开发过程中使用一些简单好记的就可以了，例如 3000,5000 等没什么含义的端口号

# 6. 响应内容类型

1. 在服务端默认发送的数据，其实是 utf-8 编程的内容，但浏览器不知道你是 utf-8 编码的内容

2. 浏览器在不知道服务器的编码的情况下会按照当前的操作系统的默认编码去解析（除了可以用 Content-Type 来指定编码，也可以在 HTML 页面中通过 meta 元素来声明当前文本的编码格式，浏览器会识别它）

3. 中文默认操作系统是 gbk

4. 解决方法就是正确告诉浏览器发送的内容是什么编码

5.Content-Type 内容类型

  • 不同资源对应的的 Content-Type 是不一样的

  • Content-Type 内容类型 http://tool.oschina.net/commons

  • 图片不需要知道编码

  • 一般只为字符数据才指定编码

# 7. 服务端渲染和客户端渲染

服务端渲染说白了就是在服务端使用模板引擎，模板引擎最早诞生于服务端，后来才发展到客户端

服务端渲染和客户端渲染的区别：

  • 客户端渲染不利于 SEO 搜索引擎优化
  • 服务端渲染是可以被爬虫抓取到的，客户端异步渲染是很难被爬虫抓取到的
  • 一个网页既不是纯异步也不是纯服务器端渲染出来的，是两者结合来做的
  • 例如京东的商品列表就是采用的是服务器渲染，目的就是为了 SEO 搜索引擎优化，而它的评论列表为了用户体验，而且不需要 SEO 优化，所以用客户端渲染

# 8.2.3 导出原理解析

exports 是 module.exports 的一个引用

exports 和 module.exports 的区别：

  • 每个模块中都有一个 module 对象
  • module 对象中有一个 exports 对象
  • 我们可以把需要导出的成员挂载到 module.exports 接口对象中去（module.exports.xxx = xxx）
  • 但是每次 module.exports.xxx 的方式很麻烦，所以 Node 为了方便在每一个模块中提供了一个成员叫： exports
  • 所以 module.exports.xxx = xxx 的方式完全可以写成：exports.xxx = xxx
  • 当一个模块需要导出单个成员的时候，这个时候必须使用： module.exports = xxx 的方式，不要使用 exports = xxx 的方式
  • 每个模块最终向外 return 的是 module.exports ，而 exports 只是 module.exports 的一个引用

# 8.2.4require 方法加载规则

  • 优先从缓存加载，再次加载只会获得接口对象，不会重新在执行一次

第三方模块

  • 凡是第三方模块都必须通过 npm 来下载，使用的时候就可以通过 require（‘包名’）的方式来进行加载使用
  • 不可能有任何一个第三方包和核心模块的名字是一样的
  • 加载规则
    • 先找到当前文件所处目录的 node_modules 目录
    • 找到 node_modules / 包名 /package.json 中的 main 属性，main 属性中就记录了包的入口模块
    • 如果 package.json 中不存在或者 main 指定的入口模块没有，则 node 会自动找该目录下的 index.js
    • 如果以上任何一个条件都不符合，则会进入上一级目录的 node_modules 目录查找
    • 一直查找到当前磁盘的 node_modules 目录，还是没有就会报 can not find module 'xxx'

# 9.2package.json

可以通过 npm init 命令来新创建一个 package.json 文件，对于咱们目前来讲，最有用的就是 dependencies 选项，可以帮我们保存第三方包的依赖信息

如果的 node_modules 删除了也不用担心，我们只需要： npm install 就会自动吧 package.json 中的 dependencies 中的所有依赖都下载回来

  • 建议每个项目的根目录都有一个 package.json 文件
  • 建议执行 npm install 包名的时候都加上 --save 这个选项，目的是用来保存依赖信息

# 9.3package.json 和 package-lock.json

npm 5 以前是不会有 package-lock.json 这个文件的，npm5 以后才会加入这个文件

当你安装包的时候，npm 都会生成或者更新 package-lock.json 这个文件

  • npm5 以后的版本安装包的时候不需要加 --save 参数，他会自动保存依赖信息
  • 当你安装包的时候，会自动创建或者是更新 package.json 这个文件
  • package-lock.json 这个文件会保存 node_modules 中所有包的信息（版本，下载地址）
    • 这样的话重新 npm install 的时候的速度就会快很多
  • 从文件来看，有个 lock，称之为锁，用来锁定版本，如果项目依赖了 1.11.1 版本，重新 install 其实会下载最新版本，package-lock.json 这个文件的另一个作用就是锁定版本号，不会自动升级
