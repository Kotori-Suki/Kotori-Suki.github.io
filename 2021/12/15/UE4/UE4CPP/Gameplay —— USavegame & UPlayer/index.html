# Gameplay —— USavegame & UPlayer

简介：这是咱 Gameplay 源码解析的最后一篇了。"Savegame" 是用来将游戏运行时数据保存到磁盘中的 "UObject" 类，主要利用了 "UObject" 的序列化与不同平台对 "ISaveGameSystem" 接口中读写磁盘操作的具体实现。而 "UPlayer" 则是代表玩家的基类，与玩家控制器相关联，其子类分别有 "ULocalPlayer"(本地玩家) 和 "UNetConnection"(网络玩家)。那为什么要放在一起说呢？因为实在不知道怎么归类啊，亲。于是就放到最后一篇，直接归类为 "杂" (bushi

# USaveGame

咱直接看源码：

![USaveGame](D:\blog\source_posts\UE4\UE4CPP\Gameplay —— USavegame & UPlayer\USaveGame.png)

Surprise！如我们所见，"USaveGame" 就只是一个支持反射的空 "UObject" 类。但其实又不完全是空的，因为咱还有注释，注释才是打开 "USaveGame" 的钥匙，让我们先看看 (翻译) 虚幻对 "SaveGame" 是怎么说的。

此类作为 SaveGame 对象的基类，可以用来保存关于游戏的状态。使用方法 如下：

  1. 创建 SaveGame 的子类时，你可以将想要保存的信息作为成员变量添加给此类。
  2. 然后 当你想保存游戏时，使用 "CreateSaveGameObject" 创建一个该 (类) 对象的实例，填充其数据，并调用 SaveGameToSlot (需要自定义 Slot 名) 进行保存。
  3. 读取游戏则只需要调用 "LoadGameFromSlot" (需要保存时的 Slot 名)，并从返回对象中读取数据即可。

接下来是 "USaveGame" 类内部的注释，也就是使用 SaveGame 的五个函数。我们分别来看看是怎么实现的

可以看出，"USaveGame" 这个类跟 "ISaveGameSystem" 的关系密切，几乎所有操作与其相关 (除了创建)。"ISaveGameSystem" 实际上只是统一与磁盘的交互接口，其子类对应不同的平台，实现不同的 "SaveGame"，"LoadGame"，"DeleteGame" 等接口。至于不同平台的具体实现，有兴趣的读者请自行研读源码 😃

# UPlayer

"UPlayer" 代表玩家，与 "PlayerController" 一一对应，其子类有 "ULocalPlayer" 和 "UNetConnection"，分别表示本地玩家和网络玩家。"ULocalPlayer" 只存在客户端，主要用于分屏游戏，而 "UNetConnection" 既存在于客户端 (一个)，又存在于服务器 (多个)，是客户端与服务器之间通信的基础。我们先看看 "UPlayer" 的实现：

可以看出，"UPlayer" 的属性不多，只有与之关联的 "PlayerController" 和 网络速度相关的整数值 (当前网络速度，配置速度)，另外还提供了运行命令的 API，以便子类使用。

"UPlayer" 就这？那当然不可能就这。接下来的两个子类才是重头戏，咱先从 "ULocalPlayer" 开始。

# ULocalPlayer

客户端中每有一个活跃的玩家都会对应有一个 "APlayerController"，同时也会对应一个 "ULocalPlayer"。如果是分屏游戏或是协助游戏 (CoopGame)，则可能存在多个 "ULocalPlayer"。另外，服务器不存在 "ULocalPlayer"。以下是 "ULocalPlayer" 的源码分析。

不难发现，"ULocalPlayer" 实际上只是在处理本地玩家的视口问题，利用 "ControllerID" 来唯一确认一个 "PlayerController"，然后确定与其关联的本地玩家。如果说 "PlayerController" 是处理玩家输入，"PlayerState" 是运行时的临时玩家数据，那 "ULocalPlayer" 则是负责将玩家显示出来 (的一部分工作)，可以将其理解成 MVC 模型，在客户端中分工明确。注意：独立服务器是不需要显示的，所以也不会有 ULocalPlayer。

# UNetConnection

与 "ULocalPlayer" 类似，作为网络连接中的 UPlayer，主要是处理客户端和服务器的数据交互，包含很多网络数据包收发操作，写者认为 "UNetConnection" 应该在网络模块单独分析，因为 "UNetConnection" 的实现是建立在其他网络相关的类上的 (如：NetDriver 等)，如果只是对 "UNetConnection" 做简单的分析，而不涉及其他网络相关的类，其实是没办法把我住虚幻的网络模块的，意义不大。所以，"UNetConnection" 的解析，我们放在后面的 NetWork 相关的博客中 (挖个坑)

结语：gameplay 的源码解读到此完结了。虽然本章的解读大多都只是对 API 的作用进行解析，没有具体分析 API 的实现，但其实 API 的实现大多都是建立在其他 API 上的，了解 API 的作用对我们理解 API 的实现也是非常有帮助的。另外知道引擎为我们提供的 API，对于客户端开发也是非常有帮助的，因为可以更大程度地利用引擎优化过的 API，以提高程序运行效率。
