# Unreal 多种指针实现原理

简介：总结虚幻中 "TSharedPtr"，"TSharedRef"，"TWeakPtr"，"TUniquePtr"，"TWeakObjectPtr" ，"TSoftObjectPtr"，"TSoftClassPtr"，"TLazyObjectPtr" 的实现原理及其区别。涉及线程安全，异步加载，引用计数等，虽然这篇难度比较低，但还是梳理一下吧。好记性不如烂笔头，顺便还能帮助一些想学的读者，就很棒 😃

# 共享指针：TSharedRef、TSharedPtr、TWeakPtr

共享指针的实现原理相信读者们也是相当熟悉了。实在不熟悉的朋友可以翻翻写者之前复刻 C++ 共享指针的博客，复刻指针支持线程安全与非线程安全模式，可以投入正常的开发使用。文章从零开始讲怎么实现一组共享指针，讲得比较详细。本节只研究 unreal 实现共享指针的原理，具体细节可能会略过一些。

我们都知道 "UE" 的共享指针有三种："TSharedPtr" , "TSharedRef" , "TWeakPtr"。俗话说：知其然，不如知其所以然。在讲原理之前，请允许我先总结一下共享指针的使用场景，以及为什么需要共享指针的一些个人看法。

# 为什么要共享指针

共享指针能帮我们更方便的管理内存，在所有共享指针都已不再指向某块内存时，程序会自动析构该内存中的对象 (如果只是简单数据类型则不需要析构)，释放内存。** 那为什么不能让 GC 来代替共享指针管理内存？又或者为什么不能用共享指针来代替 GC ？** 写者认为原因主要有以下三点：

  1. 便利性。我们都知道 "TSharedPtr" 是没办法解决循环引用的问题的，要解决这个问题，就得使用 "TWeakPtr" 。但如果我们每创建一个指针都得去考虑用 "TSharedPtr" 还是 "TWeakPtr"，那工作量简直起飞。如果使用错了，那带来的额外工作量也是让猿难以接受的。这个问题 GC 可以完美解决。那为什么还要共享指针呢？直接 GC 全部搞定不就行了？那就是第二个原因了
  2. 高效性。如果由 GC 接管对象所有引用对象的自动析构与内存释放，那 GC 对象数量将直线上升。虽然虚幻的 GC 也是一再优化，但相比智能指针，那真是天差地别。智能指针只需在改变指向的时候判断一下这个对象是否仍被引用，没有就直接析构。而 GC 还得进行可达性分析上标记，增量清除不可达对象，更新 HashTables 等，这些不可避免的步骤。
  3. 适用性。unreal 的设计是 GC 管 "UObject" 对象，以及继承了 "FGCObject" 类且重载了 "AddReferencedObjects" 函数的对象。而这些基本都是我们写游戏时无时无刻都在用的类。而对于一些原始的 C++ 类或结构体，则使用智能指针来管理，比如 Slate 就大量地用到了智能指针来自动管理内存（下一篇咱来分析一下 "Slate" 的实现原理 😃

虚幻底层还有像 内存偏移强制转换 这些没办法直接确定类型信息的操作，要用智能指针来管理这些对象 首先不说能不能实现，就算实现了操作上也非常不直观。总的来说，没有最好只有更好，我们的引擎要方便使用者使用的同时，还必须保证其拥有更好的性能。既然两者各有优缺点，那何不各取所长补其所短呢？

接下来让咱看看虚幻是怎么实现自己的共享指针的吧～～

# unreal 共享指针实现原理总览：

实现原理跟 C++ 的没什么不同，就是实现一个指针计数器类。所有指向同一个对象的共享指针，都持有同一个指针计数器 (FReferenceControllerBase) 的引用。改变指针指向时需要对该计数器中的 "Count" 进行修改，从而记录当前指向该内存的共享指针的数量。当指向某块内存的最后一个共享指针改变指向时，析构该对象释放内存。当引用控制器 (RefControllerBase) 的弱引用计数降至 0 时，析构该引用控制器。

# 弱对象指针 TWeakObjectPtr

顾名思义，就是指向 "UObject" 的弱指针。该指针不会影响对象的 GC 处理，如果引用的对象被处理掉了，则返回空指针。"TWeakObjectPtr" 不可以直接用于网络编程

开发时，我们通常会在不想影响某个对象 GC 时候选择用 "TWeakObjectPtr"

"TWeakObjectPtr" 的底层实现主要建立 "FWeakObjectPtr" 上，以下我们来分别分析 "FWeakObjectPtr" 和 "TWeakObjectPtr" 的具体实现

总结一下 "TWeakObjectPtr" 所实现的接口，以方便我们更好地使用该指针

  • IsValid ：指向的对象是否仍然有效（已初始化且未被 GC）
  • * 和 -> 运算符 ：获取对象 和 获取对象指针
  • Get ：获取对象指针
  • Reset ：重置当前弱对象指针，使其指向 nullptr
  • 构造函数 ：UObject* 直接构造，复制构造，默认构造，子类构造
  • 赋值运算符 ：UObject 赋值，子类拷贝赋值

# 软对象指针 TSoftObjectPtr

软对象指针 主要在异步加载资源时用得比较多。"TSoftObjectPtr" 保存了该对象在磁盘中的路径，是指向 UObject 的一个弱指针，不影响对象 GC。当指向的对象被加载或卸载，"TSoftObjectPtr" 将被切换为有效或挂起。

"TSoftObjectPtr" 的实现主要建立在 "TPersistentObjectPtr<>" , "FSoftObjectPath" 和 "FSoftObjectPtr" 上，我们的分析也将围绕这三个类展开。

类名说明
FSoftObjectPath记录资源路径的 FName 及 资源的子对象 FString
TPersistentObjectPtr<>"FSoftObjectPath" 的包装器，定义了一个指向 UObject 的 WeakPtr，和该对象的 "SoftObjectPath"。获取对象时，如果 WeakPtr 有效则返回，否则用 SoftObjectPath 调用 FindObject 在 Hash 中寻找该对象
FSoftObjectPtr继承于 "TPersistentObjectPtr"，主要是封装了很多对 "FSoftObject" 的操作，如获取包名，资源名，对象等
TSoftObjectPtr内部属性只有 FSoftObjectPath，是 "FSoftObjectPtr" 的模板化包装器。以及 "LoadSynchronous" 方法，同步加载资源。

# FSoftObjectPtr

"FSoftObjectPtr" 是一种指向 UObject 的弱指针，它还跟踪磁盘上对象的路径。当指向对象加载或卸载时，该指针将在有效和挂起之间来回变化，"FSoftObjectPtr" 不影响指向对象的 GC

多了一个 "LoadSynchronou" 函数，其实也只是调用了基类的 "TryLoad" 和 "Get" 方法。其他的都是在炒 "FSoftObjectPath" 的冷饭，我们就不做过多赘述了。

# TSoftObjectPtr

"TSoftObjectPtr" 其实就是对 "FSoftObjectPtr" 的包装，方法都还是一样的。包装是为了让底层代码的实现更灵活，提高代码的重用率。就好比 "TPersistentObjectPtr" 这个模板类，其实等等要讲的 "TLazyObjectPtr" 还要用到它，在收集 UObject 引用的时候也用到了它，如果我们不抽离出来，每次都得重新写一遍，这显然是不可取的。

# 懒对象指针 TLazyObjectPtr

"TLazyObjectPtr" 也是一种弱指针，其指向的 UObject 在保存时使用 GUID 创建。当被引用对象加载或卸载时，它将在 "valid" (有效) 或 "pending" (挂起) 之间来回变化。"TLazyObjectPtr" 对 GC 没有影响，不能通过网络直接使用。

"TLazyObjectPtr" 通常用于跨关卡引用，或者名字经常改变的 UObject 的引用，因为 "TLazyObjectPtr" 的查找是建立在全局唯一的 Guid 到 UObject 的映射表中的
