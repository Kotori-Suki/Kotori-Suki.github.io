# FProperty 的分析与总结

简介：本篇将对 "FProperty" 进行分析，旨在了解其内部的属性及函数的实现原理。适当地拓展分析 "FProperty" 的子类。

# 前言

相信各位也都知道，"FProperty" 其实是从 "UProperty" 中 "进化" 出来的。为什么说是进化呢？因为 "FProperty" 不再继承 "UObject"，为什么可以不再继承 "UObject" ? 因为对象和属性的区别是比较大的，我们的对象可以有很多属性，而且属性也是不需要直接 "GC" 的，可以跟随所属的对象一起 GC，所以其实 "UObject" 的很多功能，我们的 "UProperty" 是用不上的。再有就是，只要支持反射的属性都需要一个 "UProperty"，而属性在程序中又是非常之多的，所以对 "UProperty" 的 "瘦身" 必然可以减少程序运行时内存的开销，是一项不错的优化。

以下我们会对 "FProperty" 内的属性进行说明，然后再分析 "FProperty" 内的方法，及其子类的实现。当然，在此之前，我们需要先认识一下这位朋友，顺便对其 "身世" 进行一遍具体的介绍。那我们开始吧 😃

# FProperty 的介绍

"FProperty" 是一种支持反射的属性，与其说是属性，我觉得说是反射用的对象可能更直观。因为就算是简单的 "int" 型属性，如果加上 "UPROPERTY ()" 宏，引擎也会帮我们在该类的 "UClass" 里面生成一个 "FIntProperty" 对象，这个对象存着我们的 "int" 型数据的同时，还会存一些用于反射的数据，以及定义一些基于反射、使用方便的函数，"FProperty" 在引擎源码中使用广泛，像网络复制，GC 等重要功能都是离不开它的。

ClassPrivate指向类对象，表示该 "FField" 的类型
Owner该属性的 "Owner"，必须支持反射，所以一定是 "UObject" 或其子类
Next字段链表中的下一个字段节点，用于字段迭代器 "TFieldIterator"
NamePrivate字段名
FlagsPrivate字段标记，详见 "EObjectFlags"

# 属性链的初始化

咱知道了 "FProperty" 的属性，也知道每个 "UStruct" 中都会有 "PropertyLink" (属性链)，那自然就会想知道这个属性链是什么时候初始化的，怎么初始化的？答案可以在 "UClass" 的构造中找（当然也可以是 "UFunction"），因为 "UClass" 是 "UStruct" 的子类，其初始化就免不了要初始化属性链，以下我们将只对 "UClass" 属性链的初始化流程做分析。如果读者想了解 "UClass" 的整个构造过程，可以参考这篇文章。

# FProperty 常用 API

首当其冲的当然是序列化用的 "Serialize"，然后是 "PostDuplicate"，"ImportSingleProperty"..... 先列个表，把我们要分析的函数名和作用总结一下，然后再分析对应 API 的实现代码

API作用
Serialize序列化 "FProperty"
PostDuplicate将传入的 "FProperty" 的属性复制给当前 "FProperty"
ImportSingleProperty解析导入的 "FString" 参数，并将其赋值给当前 "FProperty"
ConvertFromType将指定类型的数据转换为当前 "FProperty" 类型的数据，并赋值
Identical判断两个传进来的 "void*" 型 属性数据是否相等（已计算偏移
Identical_InContainer判断两个传进来的 "void*" 型 属性容器是否相等（未计算偏移
SerializeBinProperty序列化数组等结构中的所有元素的属性值，简单类型也可以
SerializeNonMarchingBinProperty同上，但只序列化和默认值不同的元素
SerializeItem虚函数，每个子类都需要实现，用于序列化 "FProperty" 的真实值
ExportTextItem解析并将该属性的值导出为 "FString" 类型，并返回
ImportText解析导入的 "FString" 类型数据，并将其赋值给当前属性
ContainerPtrToValuePtr (多重载)返回该属性在某容器 (如: UObject*) 中的指针
IsInContainer该属性是否在某容器中
CopySingleValue将一块内存的值拷贝到另一块内存，浅拷贝，数组等类型中调用只拷贝一个元素
CopyCompleteValue将一块内存的值拷贝到另一块内存，浅拷贝，数组等类型只拷贝全部元素
ClearValue清除指定内存地址，有构造函数的对象会调用其构造函数重新初始化该内存
DestroyValue调用非 POD 类型其有析构函数的对象的析构函数（POD 类型会在 Clear 中调用 Memzero 清除内存数据
InitializeValue初始化一块内存地址，无构造函数的属性用 Memzero 申请内存，否则会调用该属性的构造函数初始化一个 Cpp 对象
GetPropertyFlags (Set / Clear)获取，设置，清空该属性的 "PropertyFlags"
GetOwner (FField 中的)获取所属的 UObject* 或 FProperty* (如果是数组元素的话)

注:

Plain Old Data (POD)
POD 指的是这样一些数据类型：基本数据类型、指针、union、数组、构造函数是 trivial 的 struct 或者 class。
POD 用来表明 C++ 中与 C 相兼容的数据类型，可以按照 C 的方式来处理（运算、拷贝等）。非 POD 数据类型与 C 不兼容，只能按照 C++ 特有的方式进行使用。

其实就是做个强制类型转换，本质上还是浅拷贝。"FArrayProperty" 的 "CopyValuesInternal" 会把所有的元素都拷贝一遍，但在 "FArrayProperty" 调用 "CopySingleValue" 只会对其一个元素进行拷贝。这两个函数是没有直接关联的，不理解可以自己翻翻源码。"CopyCompleteValue" 跟 "CopySingleValue" 的实现一样，只是会把所有元素都拷贝一遍。所以 "CopyCompleteValue" 一般用于 "FArrayProperty" 或 "FMapProperty" 这一类属性中拷贝所有元素。

# ClearValue

"ClearValue" 对于没有析构函数的对象，或者简单数据类型会直接调用 "Memzero" 来清空内存，数组类型会调用每个元素的析构函数来清除元素 (如果有析构函数)，其他类型会调用构造函数来初始化这块内存，等待 "DestroyValue" 来调用其析构函数。
