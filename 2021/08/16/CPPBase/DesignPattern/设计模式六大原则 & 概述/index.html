# 设计模式六大原则 & 概述

概述：设计模式是非常好用的程序设计小技巧，它能帮我们更好地审视自己的代码，提高代码的健壮性和可移植性。要用好一个东西，我认为首先得知道它到底为什么要这么设计，然后才是学习怎么用，所以要先了解设计模式的六大原则。

# 设计模式的六大原则

# 1、开闭原则

​ 即 对扩展开放，对修改关闭。那... 为什么要遵守开闭原则呢？道理很简单，比如说我们做游戏，每次更新都去修改之前已经测试通过的代码，为了重新测试修改后的代码的稳定性，必将带来的很多人力物力的浪费。所以我们需要对扩展开放的同时，对修改关闭。

​ 那么，怎么才能让自己设计出来的程序符合开闭原则呢？

解决办法：

  • 使用接口和抽象类，为系统定义一个相对稳定的抽象层，将不同的实现行为移至具体的实现类中完成。
  • 参数类型，引用对象尽量使用接口或者抽象类，而不是具体类；
  • 抽象层尽量保持稳定，一旦确定，不允许修改；

# 2、单一职责

​ 即 一个类只负责单一职责，不应该让所有的职责都让一个类来承担。道理也很简单，比如我们做游戏，如果让一个背包类既负责数据的存放，又负责界面的显示，那就是将这两个本来可以分开的职责耦合到了一起，这样既不利于我们对类的重用，也不利于我们后面对代码的修改（可能牵一发而动全身）

解决办法：

  • 将不同的职责封装到不同的类或模块中，一个类只负责一项职责（如 MVC 模式）

# 3、里氏代换原则

​ 即 任何基类可以出现的地方，其子类都一定能出现。这也很好理解，如果要子类都能替换基类的话，那子类就不能修改基类中原有的方法，否则替换掉就会导致程序功能发生变化。那么为什么要遵循里氏代换原则呢？因为只有这样才能真正实现基类的复用，而子类也可以扩展其功能。

解决办法：

  • 子类尽量不要重写基类的非纯虚方法，如要大量重写，可以直接继承自基类的基类（或抽象接口），使其和之前的基类是兄弟关系，而不是父子关系。

# 4、依赖倒转原则

​ 即 抽象不应该依赖于细节，细节应该依赖于抽象。高层模块不应该依赖于低层模块，两者都应该依赖其抽象。简单来说就是要建立一个抽象层，所有的细节都是抽象层具体化出来的，编程时可以是用抽象代替各种各样的细节。其中的优越性也是不言而喻：代码的重用性，健壮性，可扩展性，可移植性都是非常高的。

解决办法：

  • 建立抽象层，每个类都尽量实现其抽象类或接口
  • 参数类型，引用对象尽量使用接口或者抽象类，而不是具体类

# 5、接口隔离原则

​ 即 把不需要的接口方法隔离出来，形成新的接口，以避免接口方法的臃肿冗余。

为什么要这么做呢？

  • 首先这样做很好地减少代码冗余
  • 其次还可以实现接口高内聚，提高接口专门处理一项事物的能力
  • 实现定制服务，只提供访问者需要的方法
  • 接口设计细粒度小，代码灵活性高，可可重用性就越好。当然也不能太小，会使代码过分复杂化。

解决办法：

  • 将臃肿的接口方法分解成多个接口

  • 将不同功能的接口方法分解成多个接口

# 6、迪米特法则（最小知道原则）

​ 即 一个对象应该对其他对象有最少的了解。就是在告诉我们要注意解耦！解耦！因为只有解耦做得好，代码的细粒度才会越小，我们代码的复用性才能越好。就好比游戏中，我只想换个皮肤，你给我把血都扣光了，这自然是不行的。

解决办法：

  • 首先，没有必要 public 的，就尽量用 protected，private 设置权限；

  • 其次，可以使用友元类。即 出现在成员变量、方法的输入输出参数中的类。C++ 中用 friend 关键字来表示。

  • 最后，如果一个方法可以放在本类中，也可以放在其它类中。如果这个方法放在本类中不会增加耦合，也不会对本类产生负面影响，那就放置在本类中。

最后的最后，多说一句。之前在看博客的时候，看到过这么一句话 ——"框架才是大智慧，设计模式只是小技巧。" 确实，设计模式只是为了设计出健壮可靠的代码，而底层框架的设计需要的是长年累月沉淀下来的经验。但这并不意味着设计模式不重要，我认为这恰恰意味着设计模式是通往 “大智慧” 的起点。如果不懂设计模式，我们看框架源码的时候也很难理解其中的优越性，更别说从中学到什么 “大智慧” 了
