# UE4 插件开发

简介：因为是开发篇，所以本篇对一些函数的分析可能不会太深入。主要也是一些 Slate 语法，不了解的朋友建议先阅读这篇博客。本篇将围绕写者自制的任务编辑器展开，从无到有创建编辑器插件。中途也会插播一些问题的个人见解，比如：引擎为什么要这么实现的，这么实现的好处是什么等等。那咱开始吧 😃

# 新建一个插件

创建完之后，引擎会自动为我们生成该插件的源文件和反射文件，还会在菜单栏中创建一个按钮方便打开我们的插件，如下图：

# QuestEditorStyle

如同名字一样，该类是管理插件的风格的，包括一些 "Textures" 的读取，存储 "Style" 所用到的数据集 "SlateStyleSet" 对象的共享指针等。函数和属性对应的作用列成表格如下：

属性 / 函数作用
Initialize初始化 "FSlateStyleSet" 实例 (设置内容根目录，添加按钮的 Brush 等)，并在 "FSlateStyleRegistry" 中注册该 "StyleInstance"
Shutdown注销该 "StyleInstance"，重置 "StyleInstance" 中的数据
ReloadTextures从磁盘中重新读取所有图片资源
Get获取 "StyleInstance"
GetStyleSetName返回该编辑器的名称
Create初始化 "FSlateStyleSet" 实例 (设置内容根目录，添加按钮的 Brush 等)，必须在
StyleInstance存着该 Slate 风格的所有属性 (内容根目录，各种名字和属性的映射等)

# FQuestEditorCommands

该类继承自 "TCommands" (命令集基类)，可以看出是用来收集命令信息的。构造函数也只是简单地初始化了一下 "TCommands" 基类，"RegisterCommands" 函数用来注册 UI 命令的 (使用 UI_COMMAND 宏，注册方法下文具体分析) 。"OpenPluginWindow" 只是一个 "CommandId"，记录了打开该插件窗口的一条命令，在 "RegisterCommands" 中注册。

# 明确窗口布局

明确窗口布局其实是为了 "Slate" 的创建能更加顺利，同时可以进一步明确我们的编辑器最后支持的操作有哪些，这些都是要尽早规划好的。以下是写者画的布局图 (比较硬核)

通过上面个的图，可以看出我们这次要实现的功能有以下几点：

  • 蓝色框是菜单栏，用于添加新任务，移除任务，保存等操作
  • 绿色框是提供给我们对任务内容进行直接编辑的窗口
  • 黄色框将按 任务类型 / 等级 对已添加的任务进行分类显示

要实现上面的需求，我们最少要创建两个 Slate 类 (黄框和蓝框，绿框可以用 "SDetailsView")。

  1. 对于绿框 我们需要有一个记录任务数据的 "Struct" 。
  2. 对于粉框 我们要一个用于记录临时数据的 "TMap"，保存时写入 "DataTable" 中 ("Slate" 的编写会比较难) 。
  3. 最后是蓝框的按钮，主要是回调函数的编写。

明确完这些就可以开工了。我的习惯是先做数据结构，然后做编辑器，也就是 绿框 -> 粉框 -> 蓝框
