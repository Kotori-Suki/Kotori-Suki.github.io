# 工厂模式（Factory）

概述：如果说简单工厂模式是小农经济的小作坊，生产什么都可以在小作坊里面划一块来实现；那工厂模式就是现代化地，一个工厂只批量生产一种产品。那肯定就有人问了：只能生产一种产品？那不就是退化了的简单工厂？其实正好相反，工厂模式把工厂也抽象了出来，同时一个工厂只生产一种产品，既符合开闭原则，又不会有职责过重，代码臃肿等问题

# 特点：

  1. 由一个工厂抽象接口，一个产品抽象接口，若干具体工厂，若干具体产品组成
  2. 每个具体工厂只负责生产一个具体产品

# 优缺点：

优点：

  • 用户只需要知道具体工厂名称即可得到想要的产品，无需知道产品的具体创建过程；
  • 增强灵活性，对于新产品的创建，只需多写一个相应的工厂类；
  • 典型的解耦框架。高层模块只管用抽象类即可，无需关心其他实现类，符合开闭原则，最小知道原则，依赖倒置原则，里氏替换原则。

缺点:

  • 类的个数容易过多，增加代码结构复杂度；
  • 增加了系统的抽象性和理解难度；
  • 抽象产品只能生产一种产品（抽象工厂模式可解决此弊端）

# 应用场景：

  • 客户只知道创建产品的工厂名，而不知道具体产品名。框架的开发经常就是这样，需要维护一个抽象层
  • 创建对象的任务由多个具体子工厂中的某个完成。
  • 客户不关心创建产品的细节，只关注产品的抽象。

结语：工厂模式是比较常用的模式，特别是在开发大型程序的框架时，抽象层并不知道实现层到底会需要什么具体类，但如果用工厂模式，那就不需要知道了。因为这个工作可以交给后面实现具体层的程序员来做，他只需要用具体工厂覆盖我们的抽象工厂，即可通过抽象工厂对象调用具体工厂中重写的虚函数，实现抽象层的多态
